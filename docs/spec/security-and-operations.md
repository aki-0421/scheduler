# Security And Operations

Codex Scheduler runs local AI work against local files. The implementation treats scheduling, path trust, capability tokens, sandbox configuration, and audit logs as product-visible safety controls.

## Trust Boundary

Trusted components:

- The user account running the app.
- The Tauri app bundle.
- The bundled `codex-schedulerd` sidecar.
- The bundled `codex-schedule` sidecar.
- The SQLite database under the app data directory.

Conditionally trusted components:

- User-trusted project paths.
- The configured Codex binary.
- Git.

Untrusted inputs:

- Prompts and model-generated follow-up requests.
- CLI arguments generated by scheduled Codex sessions.
- Repository contents, command output, and external data consumed during runs.

## Project Trust

Repository-backed tasks are allowed only under trusted project roots. Project trust stores canonical path information, Git metadata when available, default branch when detectable, and a `trustedAt` timestamp.

Removing trust clears the project trust timestamp and records an audit event. Existing tasks are not deleted or automatically paused, but they may fail until a valid trusted path exists again.

Scheduled Codex sessions cannot trust or untrust projects.

## Path Opening Policy

The desktop backend restricts `open_path` to scheduler-owned logs, scheduler-owned worktrees, scheduler-owned chat workspaces, or trusted project roots. This prevents arbitrary path opening through UI-provided strings.

## Capability Tokens

Scheduled runs receive a run-scoped token only when schedule CLI access is enabled. The database stores token hashes, not raw token values.

Token records include:

- run ID,
- task ID,
- capability list JSON,
- expiration,
- max create count,
- current create count,
- revocation timestamp.

The daemon enforces token capability checks for scheduled-run RPC calls. Implemented capability names include schedule creation, current-task updates, any-task updates, current-task pause, and run-now access.

## Sandbox And Approval Policy

Sandbox modes:

- `read-only`
- `workspace-write`
- `danger-full-access`

Approval policies:

- `never`
- `on-request`
- `untrusted`

The frontend default is read-only with never-ask approval. Scheduled runs avoid interactive approval stalls by executing with `approval_policy="never"` even if a task stores another approval policy; the runner records that override as a warning. `danger-full-access` requires explicit task confirmation in the UI and an explicit runner allowance before execution.

## Audit

Task create/update/delete/pause/resume/run-now and project untrust actions record audit events with actor type, optional actor ID, action name, reason, and before/after JSON where applicable.

Actor types distinguish user, daemon, CLI, and scheduled-run actions.

## Diagnostics

The daemon exposes health and diagnostics over RPC. Diagnostics include app version, schema version, data directory, socket path, DB/log sizes, task/run counts, scheduler enabled state, Codex path existence, tick interval, and last tick time.

The desktop app can export diagnostics with daemon health, daemon diagnostics, redacted daemon log tails, OS version, and timestamp. Log tail export redacts sensitive patterns such as API keys and run tokens.

## Notifications

The desktop backend polls run status snapshots and sends desktop notifications for failed and timed-out runs when notifications are enabled.

## Cleanup And Retention

Retention cleanup deletes expired capability tokens, old terminal run history, eligible old logs, and eligible worktrees. Running runs are not removed as old history. Dirty worktrees are skipped during delete-after-days cleanup.

## Release Operations

The release artifact is a Tauri `.app` bundle with sidecar binaries. Release builds must prepare sidecars before packaging and sign/notarize the full app bundle, including sidecars.

Recommended verification after release build:

```bash
codesign --verify --deep --strict --verbose=2 apps/desktop/src-tauri/target/release/bundle/macos/Codex\ Scheduler.app
spctl --assess --type execute --verbose apps/desktop/src-tauri/target/release/bundle/macos/Codex\ Scheduler.app
```

Use the release document for the full macOS signing and notarization checklist.

## Pull Request Operations

When creating or updating a GitHub pull request for this repository, use `.github/PULL_REQUEST_TEMPLATE.md` as the required body structure. Preserve the template headings and fill in each section rather than replacing it with an ad hoc summary.

If a PR is created through `gh pr create`, either let GitHub apply the template or pass a body that follows the template exactly. If the body must be corrected later, update it to the same structure before handing off the PR.

## Verification Commands

Use these checks after implementation changes:

```bash
cargo test --workspace
pnpm lint
pnpm test
pnpm --filter desktop build
```

Use `agent-browser` for UI behavior verification and save screenshots in `/tmp` or an ignored local `tmp/` directory.
